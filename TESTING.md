# Running anchord's tests

anchord ships with three test layers and a release gate. The
**Test report** block at the bottom of [README.md](README.md) is
auto-generated by running the full suite — that block is what the
release pipeline checks before letting a tag through.

## TL;DR — update the test report

The only requirement on the host is **Docker**. Everything else
(Go, bash, git, the e2e runner) is loaded into a transient Alpine
container. Identical behaviour on Windows, macOS, Linux, and CI.

| Host           | Command                                    |
|----------------|--------------------------------------------|
| Linux / macOS  | `bash scripts/update-test-report.sh`       |
| Windows        | `pwsh scripts/update-test-report.ps1`      |
| CI (any OS)    | same as Linux                              |

A green run rewrites the `<!-- TEST-REPORT-START -->` block in
README.md with the current code hash, Go test summary, and per-scenario
e2e results. A red run leaves README.md untouched.

Commit the README.md change before tagging.

## What it actually runs

1. `scripts/code-hash.sh` — deterministic SHA-256 over `*.go`,
   `go.mod`, `go.sum`, `Dockerfile`, `test/`, and `scripts/`. Excludes
   docs (README, ARCHITECTURE, SPEC, CONTEXT, CLAUDE.md), `LICENSE`,
   the example compose, and CI workflows. The hash is what the release
   gate compares.

2. `go vet ./...` and `go test -count=1 ./...` inside `golang:1.25-alpine`.

3. The e2e harness (`test/e2e/up.sh`) across all four DHCP scenarios:
   `v4-only`, `v6-only`, `both`, `none`. Inside the harness:
   - boots a per-scenario stack (anchord + dnsmasq + service-anchor +
     listener + probe) on internal Docker bridges,
   - asserts macvlan / nftables / DNAT-map state (Phase 1),
   - drives a probe through the inbound DNAT path and verifies source
     IP preservation, reconciler convergence after a service-anchor
     recreate, and graceful shutdown (Phase 2 — S-2, S-3, S-6).

## Environment knobs

| Variable               | Default | Meaning |
|------------------------|---------|---------|
| `E2E_BRIDGE_FLOOD_FIX` | `1`     | On Docker Desktop, sets `bridge-nf-call-iptables=0` host-wide so v4 DHCP traffic isn't dropped by Docker's iptables FORWARD rules. Set to `0` on a real Linux host with a physical VLAN parent. |
| `WAIT_SECONDS`         | `25`    | How long the harness sleeps between `compose up` and assertions. Increase if your machine is slow. |
| `NO_TEARDOWN`          | `0`     | Set to `1` to leave the per-scenario stack running after assertions for manual inspection. |

## Verifying the gate locally

The release pipeline runs `scripts/verify-test-report.sh` on every
tag push. To pre-flight before tagging:

```sh
# Linux / macOS
bash scripts/verify-test-report.sh

# Windows (PowerShell)
docker run --rm -v "${PWD}:/repo" -w /repo `
    alpine:3.21 sh -c "apk add -q --no-cache bash git >/dev/null && bash scripts/verify-test-report.sh"
```

A `0` exit means the README's recorded hash matches the current source.
Anything else means you have uncommitted changes (or someone hand-edited
the test report block); re-run `update-test-report.sh` and commit.

## CI workflows

- `.github/workflows/ci.yml` runs `go vet` + `go test` on every push
  and PR. Informational; never blocks.
- `.github/workflows/release-gate.yml` runs on every `v*` tag and
  blocks the release unless **both** of:
    1. the tagged commit is reachable from `main` (tags on feature
       branches never release), and
    2. README.md's recorded test-report hash matches the current
       source.

The gate is intentionally trust-based and cheap: it does not re-run
the e2e harness in CI (Docker-in-Docker on GitHub-hosted runners is
flaky and slow). Instead, it trusts that
`scripts/update-test-report.sh` only ever writes to README.md after a
green run, so a matching hash means a matching successful run.

### Recommended branch protection on `main`

The gate guarantees release-readiness for the *commit being tagged*.
To make the gate maximally useful, also configure GitHub branch
protection on `main`:

- Require pull requests before merging.
- Require status checks (`CI / go-tests`) to pass before merge.
- Disallow direct pushes to `main`.

That way every commit on `main` has at least passed `go vet` + `go test`,
and a release tag on `main` reflects an end-to-end-verified state via
the test report block.

## Running just one piece

| Want                              | Command |
|-----------------------------------|---------|
| `go test ./...` only              | `docker run --rm -v "$(pwd):/src" -w /src golang:1.25-alpine go test ./...` |
| The integration tests (nftables) | `bash test/integration/run.sh` (or `.ps1`) — needs `--cap-add=NET_ADMIN` |
| One e2e scenario                  | `WAIT_SECONDS=20 bash test/e2e/up.sh both` (or `.ps1`) |
| Just the code hash                | `bash scripts/code-hash.sh` |
